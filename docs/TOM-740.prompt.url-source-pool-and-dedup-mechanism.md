## 🔧 具体要求与约束 (Requirements & Constraints)

### 技术方向
- **首选存储**：SQLite
- **语言**：Node.js
- **目标**：本地轻量、结构清晰、可直接供 runner 使用

### 不要做的事
- 不要引入远程数据库
- 不要引入服务端 API
- 不要引入复杂后台系统
- 不要做多用户协作平台
- 不要用 Markdown 作为主数据层
- 不要为了“未来扩展”过度抽象

### 必须解决的问题
1. URL 存储
2. URL 唯一约束
3. URL 来源记录
4. URL 状态管理
5. runner 取下一条待执行 URL
6. 标记 URL 已处理 / 失败 / 跳过

---

## 🧱 建议的数据模型 (Minimum Data Model)

建议至少有一张主表，例如：`source_urls`

建议字段：
- `id` INTEGER PRIMARY KEY
- `url` TEXT NOT NULL UNIQUE
- `canonical_url` TEXT (可选，若你要做标准化)
- `source_type` TEXT NOT NULL
- `source_name` TEXT
- `content_type` TEXT
- `status` TEXT NOT NULL DEFAULT 'pending'
- `first_added_at` TEXT NOT NULL
- `last_processed_at` TEXT
- `last_run_id` TEXT
- `notes` TEXT

### 状态枚举
至少支持：
- `pending`
- `processed`
- `failed`
- `skipped`

## 🧪 测试策略 (Testing Strategy)

### 重点测试内容
优先测试这些内容：
- URL 唯一约束是否生效
- 重复插入同一 URL 是否被阻止或幂等处理
- pending → processed / failed / skipped 状态流转是否正确
- 获取下一条待执行 URL 是否稳定
- 处理后 URL 是否不会被误再次取出

### 测试要求
- 如果写测试，重点写在数据层和状态流转层
- 不要为了 SQLite 任务引入过重测试基建
- 测试服务于正确性，不服务于形式主义
