## 🔧 具体要求与约束 (Requirements & Constraints)

### 1. 触发机制 (Trigger)
- 必须集成在 `run-daily.sh` 或主执行逻辑的末尾。
- 无论执行状态是 `success`、`timeout` 还是 `failure`，都必须触发回传。

### 2. 频道回传规格 (Discord Notification Spec)
推送的消息必须包含以下结构化信息：
- **执行状态**：✅ 成功 / ❌ 失败 / ⚠️ 超时
- **任务信息**：Run ID, 目标 URL, 耗时 (Duration)
- **内容指标**：Article 字符数, Tweets 数量
- **审核结论**：Pass / Warning / Fail (由逻辑初步判定或占位)
- **证据路径**：本地绝对路径 (Evidence Pack Path)

**实现约束（补充）**：
- Discord 推送优先使用既有 OpenClaw 体系（`openclaw message send` / 等效 `sessions_send` 能力）。
- `DISCORD_SCRIPT_SNAP_EXEC_WEBHOOK_URL` 仅作为兜底 fallback，不作为主路径前置要求。
- 若 OpenClaw 与 Webhook 均不可用，脚本必须给出明确 warning，但不能阻断 daily run 主流程。
- 频道 ID `1489176810410479696` 应出现在消息正文中，便于对账。

### 3. 审核归档规格 (Review Archival Spec)
- **存放位置**：必须保存在当前 Run 的输出目录下，例如 `outputs/YYYY-MM-DD/run-xxx/review-summary.md`。
- **禁止事项**：**禁止**将此类执行审核内容写入 Obsidian 项目笔记库，它属于执行证据，不属于长期知识资产。
- **内容要求**：
  - 术语准确性初评（规则层）。
  - 结构化组件（Table/Callout）检测（规则层）。
  - 提取内容是否完整（是否截断/为空）（规则层）。
  - 基于 OpenClaw Agent 的语义可行性评审（模型层），至少输出：Review Level、Feasibility、Risks、Recommendations、Model 标识。

### 4. 技术实现建议
- 使用 `curl` 或已有的 `openclaw sessions_send` 接口进行 Discord 推送。
- 使用 Node.js 脚本解析 `evidence-pack.json` 并生成 Markdown 格式的审核报告。
- 在 `run-daily.sh` 中通过 `trap` 或显式尾段，保证 post-run hook 在各种退出状态都执行。
- 若可行，补充 run 级别元信息（如 `durationMs`），避免由日志反推耗时。
